Summary

The Import files set the structure of the Ace Databases.
There are 4 main files of which 3 are required:

  1. concepts.txt - required
  2. descriptions.txt - required
  3. relationships.txt - required
  4. ids.txt - optional

concepts.txt

A required file containing all the concepts of the terminology database, and their status and version information.

Fields

Name Type Notes
concept uuid UUID
Definition The UUID of the status concept associated with this record.
Coding
  • See : Generating a UUID
  • Example: 
    concepts.append(Type5UuidFactory.get(ReadStatics.READV3_CONCEPT_ID,conceptparts[readcodeIndex]).toString());
status uuid UUID
Definition The UUID of the concept's status concept.
Coding See: Getting a Status UUID
primitive Boolean
Definition A boolean value that determines if the concept is primitive or defined.
This is used to indicate if the codesystem is a description logic terminology i.e. if the terminology defines it's structures. For example if in geometry a triangle has to have 3 sides and this was defined as part of the structure then this boolean would be false (i.e. defined).
If however the terminology just organises the terms and does not define them then this would be true (i.e. primitive)
The value is represented by a string 0 (false == defined) or 1 (true == primitive)
Coding Assume 1 unless you know it's defined
effective date Date
Definition The date this change became, or will become, effective, with a required granularity of seconds.
Coding See: Effective Date
path uuid UUID
Definition The UUID of the development path upon which this record is committed.
Coding See: PathUUID

descriptions.txt

A required file containing all the descriptions of the terminology database, and their status and version information.

Fields

Name Type Notes
description uuid UUID
Definition The UUID of the description.
Coding
  • See : Generating a UUID
  • Example: 
    UUID descUuid = Type5UuidFactory.get(genPrefix,termparts[termidIndex] + term);
descriptions.append(descUuid.toString());
status uuid UUID
Definition The UUID of the description's status concept.
Coding See: Getting a Status UUID
concept uuid UUID
Definition The UUID of the description's concept.
Coding
  • See : Generating a UUID
  • Example: 
    descriptions.append(Type5UuidFactory.get(ReadStatics.READV3_CONCEPT_ID,
termidReadCodeMap.get(termparts[termidIndex])).toString());
term String:32K
Definition The description's term, represented as a string, up to 32,000 characters, and can contain xml, xhtml, or html markup. Currently, only html or xhtml will be rendered according to the Java html rendering capabilities. Future version of the ACE editor may support the association of an XSLT style sheet with a description type, allowing dynamic rendering of xml content.
Coding Simply append the relevant string into the file e.g. 
descriptions.append(term);
capitalization status Boolean
Definition A boolean value that represents the capitalization status of the term. The semantics of this value is specific to the terminology system being imported. Future version of the ace environment may replace this value with a concept enumeration. The value is represented by a string 0 (false == defined) or 1 (true == primitive)
Coding Assume 1 unless you know different
description type uuid UUID
Definition The UUID of the description's type.
The description type is used to validate the description e.g. the length of text allowed or a given pattern.

When to create a new description type and when to reuse an exisiting one:
  • If an existing description type is more general then you can reuse it while attaching new validation patterns.
    For example if a descriptiption had a rule:
    "length must be less than or equal to 30 Chars and must start with a capital letter"
    and there is an existing description which only allows a 30 Char length you can use that and then apply a pattern validation making sure that it starts with a capital letter
  • If the rules are different from existing description types (for instance you need a max length of 199 chars) then you would need to create a new one in ArchitectonicAuxiliary
Coding
  • See Getting a Concept UUID
  • Example: 
    descriptions.append(descriptionTypeConcept.getUids().iterator().next().toString();
language code String
Definition The description's language code, represented as a string. SNOMED typically assumes these language codes are ISO 639- 1 codes, but any string representation is accepted by the ACE Editor. Future version of the ace environment may replace this value with a concept enumeration.
Coding Usually simply append the relevant string e.g. 
descriptions.append("en-gb");
effective date Date
Definition The date this change became, or will become, effective, with a required granularity of seconds.
Coding See: Effective Date
path uuid UUID
Definition The Universally Unique Identifier of the development path upon which this record is committed.
Coding See: PathUUID

relationships.txt

A required file containing all the relationships of the terminology database, and their status and version information.

Fields

Name Type Notes
relationship uuid UUID
Definition The UUID of the relationship.
Coding
  • See : Generating a UUID
  • Example: 
    relationships.append(Type5UuidFactory.get(ReadStatics.READV3_REL_ID,
relationshipParts[childReadcodeIndex]+relationshipParts[parentReadcodeIndex]).toString());
status uuid UUID
Definition The UUID of the relationship's status.
Coding See: Getting a Status UUID
source concept uuid UUID
Definition The UUID of the relationship's source concept.
The Source concept is where the relationship starts/originates. What the source and destination concept directions mean depends on the relationship. For example:
  • If an IS_A relationship the the source would be the CHILD and the destination would be the Parent
  • If an AssociatedFindingSite relationship then the relationship is horizontal.
Coding
  • See : Generating a UUID
  • Example: 
    relationships.append(Type5UuidFactory.get(ReadStatics.READV3_CONCEPT_ID,
relationshipParts[childReadcodeIndex]).toString());
relationship type uuid UUID
Definition The UUID of the relationship's type.
The relationship types can be found in ArchitectonicAuxiliary following a text search for: 
IS_A_REL(PrimordialId.IS_A_REL_ID, new String[] {"is a (relationship type)"}, null,
new I_ConceptualizeUniversally[] { RELATIONSHIP }),
The meanings of the existing SNOMED relationship types can be found in the relevant SNOMED guides for example:

SNOMED_CT_Technical_Implementation_Guide_20090131.pdf

If a relevant relationship type dies not exist then create one in ArchitectonicAuxiliary
Coding
  • See Getting a Concept UUID
  • Example: 
    relationships.append(ArchitectonicAuxiliary.Concept.IS_A_REL.getUids().iterator().next().toString());
destination concept uuid UUID
Definition The UUID of the relationship's destination concept.
The destination concept is where the relationship finishes/ends.
This would point to the PARENT in an IS_A relationship
Coding
  • See : Generating a UUID
  • Example: 
    relationships.append(Type5UuidFactory.get(ReadStatics.READV3_CONCEPT_ID,
relationshipParts[parentReadcodeIndex]).toString());
characteristic type uuid UUID
Definition The UUID of the relationship's characteristic type concept.
The relationship types concepts can be found in ArchitectonicAuxiliary following a text search for:
STATED_RELATIONSHIP
Coding
  • See Getting a Concept UUID
  • Example: 
    relationships.append(ArchitectonicAuxiliary.Concept.STATED_RELATIONSHIP.getUids().iterator().next().toString());
refinability uuid UUID
Definition The UUID of the relationship's refinability concept.
The refinability concepts can be found in ArchitectonicAuxiliary following a text search for:
RELATIONSHIP_REFINABILITY
It is used to define if for this relationship is it appropriate to refine this.
For example:
If the user must enter more data to complete this relationship then this value would be the UUID for MANDATORY
Coding
  • See Getting a Concept UUID
  • Example: 
    relationships.append(ArchitectonicAuxiliary.Concept.NOT_REFINABLE.getUids().iterator().next().toString());
relationship group Integer
Definition An integer, represented as a string, that desiginates which relationship group this relationship is part of.
A designation of 0 indicates that this relationship is not part of a designated relationship group.
If the value is 2 or more then the concepts all relate to the same instance.

For example: A single disease affects 2 other concepts e.g the lungs and the heart.

Rule:An IS_A relationship is never grouped. i.e. if the relationship type is IS_A then this value is 0.
Coding
  • Assume 0 unless you know better
  • Example: 
    relationships.append("0");

    ids.txt

    An optional file that provides history information about identifiers and also allows representation of alternate UUIDs for a component, as well as alternate representation forms (snomed ids, loinc ids, oids) for identifiers.

    Fields

    Name Type Notes
    primary uuid UUID
    Definition The primary UUID of a component.
    Coding See : Generating a UUID
    source system uuid UUID
    Definition The UUID that designates the source system (SNOMED ID, UUID, OID) of the identifier. The identifier source must come from one of the concepts in the Architectonic Concept->identifier source, otherwise the native data type will simply be represented as a string.
    Basically "How do you want to describe it" e.g. ReadV2, Read4Byte,SNOMED etc.
    The source system concepts can be found in ArchitectonicAuxiliary following a text search for: 
    ID_SOURCE("identifier source",
    If an identifier source for your system is not in the laist then you will need to add it.
    Coding 
    ArchitectonicAuxiliary.Concept.SNOMED_INT_ID.getUids().iterator().next().toString();
    source id String
    Definition A string that represents the identifier. This string may be an oid, UUID, SNOMED Id, LOINC id, or any other representation as determined by the source. Depending upon the type of identifier, this id may be converted into a native type internally, or be retained as a String internally. The environment will use the source uuid field to determine what native data type will be used to represent the identifier.
    Coding 
    ids.append(termidReadCodeMap.get(termparts[termidIndex])).toString());
    status uuid UUID
    Definition The UUID of the status concept associated with this identifier record.
    Coding See: Getting a Status UUID
    effective date Date
    Definition The date this change became, or will become, effective, with a required granularity of seconds.
    Coding See: Effective Date
    path uuid UUID
    Definition The Universally Unique Identifier of the development path upon which this record is committed.
    Coding See: PathUUID

    illicit_words.txt

    An optional file consisting of a single column, containing records of words that are not permitted for use in descriptions. The illicit words list is available though the ACE api, and can be used to provide algorithmic quality assurance of content that is filtered through a workflow process.

    Fields

    Name Type Notes
    illicit word String A word not permitted for use in descriptions

    licit_words.txt

    An optional file consisting of a single column, containing records of words that are acceptable for use in descriptions. The licit words list is available though the ACE api, and can be used to provide algorithmic quality assurance of content that is filtered through a workflow process.

    Fields

    Name Type Notes
    licit word String A word acceptable for use in descriptions
